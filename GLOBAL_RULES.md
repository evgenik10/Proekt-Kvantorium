# Глобальные правила проекта: AI-диктофон на M5StickC Plus2

## 1) Цель и границы проекта
- Устройство: **M5StickC Plus2** (UIFlow2 MicroPython).
- Назначение: записывать речь, отправлять в облако, получать:
  1. исходную аудиозапись,
  2. транскрибацию,
  3. краткое резюме «факты/договорённости/задачи»,
  4. доставку результата в Telegram.
- Критичный принцип: **устройство — тонкий клиент, ИИ-обработка — на сервере**.

---

## 2) Базовая архитектура (обязательно)

### 2.1 Компоненты
- **Edge (M5StickC Plus2)**:
  - захват аудио через `audio.Recorder`,
  - подключение к Wi‑Fi через `network.WLAN(network.STA_IF)`,
  - отправка файла/чанков на backend.
- **Backend API (ваш сервер, Python/Node/Go)**:
  - принимает аудио от устройства,
  - вызывает Silero STT,
  - вызывает Gemini для структурирования текста,
  - отправляет в Telegram.
- **Внешние API**:
  - Silero API — распознавание (и при необходимости TTS/доп. обработка),
  - Gemini API — LLM-анализ и суммаризация,
  - Telegram Bot API — доставка результатов.

### 2.2 Почему именно так
- На микроконтроллере нельзя безопасно и удобно хранить все ключи и делать сложную оркестрацию.
- Централизация логики в backend упрощает обновления, мониторинг и контроль стоимости.

---

## 3) Жёсткие правила по API (по изученным источникам)

### 3.1 UIFlow MicroPython (устройство)
- Для записи использовать модуль `audio` (`Recorder`).
- Для сети использовать STA-режим: `network.WLAN(network.STA_IF)`, проверять `wlan.isconnected()` до любой отправки.
- Ошибки Wi‑Fi не игнорировать: обязательны повторные попытки и явный таймаут.

### 3.2 Silero API (STT/TTS)
- Для транскрибации использовать endpoint `POST /transcribe`.
- В `payload` передавать **base64** бинарного аудио.
- Ограничение длины аудио: **< 1024 секунд**.
- Держать совместимые форматы: `raw`, `wav`, `opus`, `webm/opus`.
- В боевом режиме передавать `remote_id` для трассировки каждого запроса.

### 3.3 Gemini API (LLM обработка)
- Ключ задавать в переменной окружения `GEMINI_API_KEY` на backend.
- Базовый метод — `generateContent` (модель класса flash для быстрого резюме).
- Промпт должен требовать **строгий JSON-ответ** без свободного текста.

## 4) Единый контракт данных (обязательно)

### 4.1 Вход backend от устройства
```json
{
  "device_id": "m5plus2-001",
  "session_id": "uuid",
  "started_at": "2026-03-28T12:00:00Z",
  "audio_format": "wav",
  "sample_rate": 16000,
  "channels": 1,
  "audio_base64": "..."
}
```

### 4.2 Выход backend (для Telegram и хранения)
```json
{
  "session_id": "uuid",
  "transcript": "...",
  "summary": "...",
  "facts": ["..."],
  "decisions": ["..."],
  "tasks": [
    {
      "task": "...",
      "owner": "...",
      "due_date": null,
      "priority": "medium"
    }
  ],
  "risks": ["..."],
  "audio_file_url": "https://.../session.wav"
}
```

---

## 5) Правила суммаризации (LLM policy)
- Всегда выделять минимум 4 блока:
  1. `facts` (обсуждённые факты),
  2. `decisions` (принятые решения),
  3. `tasks` (кто/что/когда),
  4. `open_questions` (что не решено).
- Если данных недостаточно — писать `unknown`, не выдумывать.
- Нельзя смешивать цитаты и интерпретации без пометки.

---

## 6) Telegram-правила доставки
- Отправка в Telegram выполняется **только backend-сервисом**.
- Порядок сообщений:
  1. аудиофайл,
  2. полный transcript,
  3. структурированное summary (facts/decisions/tasks).
- При длине транскрипта > лимита Telegram — отправлять частями с нумерацией.

---

## 7) Безопасность и секреты
- НИКОГДА не хранить `api_token` Silero, `GEMINI_API_KEY`, `BOT_TOKEN` Telegram в прошивке и репозитории.
- Все секреты — только в backend `.env`/secret manager.
- Логи не должны содержать полный `audio_base64` и ключи.
- Обязательно использовать HTTPS при обмене устройство ↔ backend.

---

## 8) Надёжность и отказоустойчивость
- Ретраи с экспоненциальной паузой для сетевых ошибок.
- Идемпотентность: `session_id` уникален; повторная обработка не должна дублировать Telegram-уведомления.
- Таймауты:
  - upload с устройства,
  - вызов Silero,
  - вызов Gemini,
  - отправка в Telegram.
- При частичном успехе (например, есть transcript, но нет summary) — отправлять доступный результат и статус ошибки.

---

## 9) Качество результата (Definition of Done)
Сессия считается успешной, если:
- аудио сохранено,
- транскрипт получен,
- JSON-суммаризация валидна по схеме,
- сообщение отправлено в Telegram,
- в логах есть `session_id`, длительность и коды ответов API.

---

## 10) Наблюдаемость (минимум)
- Метрики:
  - `stt_latency_ms`, `llm_latency_ms`, `telegram_latency_ms`,
  - `stt_error_rate`, `llm_error_rate`, `delivery_error_rate`.
- Логи:
  - `session_id`, `device_id`, размер аудио, длительность обработки, итоговый статус.

---

## 11) Ограничения первой версии (V1)
- Язык: RU (EN как optional).
- Канал аудио: mono.
- Максимальная длительность сессии: безопасно держать ниже лимита Silero (<1024 сек), целевой рабочий лимит V1: **до 10 минут**.
- Обработка только по кнопке (push-to-record), без постоянного «прослушивания». 

---

## 12) Правила развития проекта
- Любое изменение API-контракта — только с bump версии схемы.
- Любая новая ИИ-фича (NER, action items auto-assign, календарь) добавляется за feature-flag.
- Перед релизом проверять совместимость прошивки UIFlow2 и backend-контракта на тестовой сессии.
